## 1. 시작하기

### UMG란?

Universal MCP Gateway(UMG)는 기업 내 다양한 도구(n8n 워크플로우, Cube.js 분석, AWS MCP 서버)를 단일 MCP(Model Context Protocol) 엔드포인트로 통합하는 중앙화 미들웨어입니다.

AI 에이전트(Cursor, Claude Desktop 등)가 UMG를 통해 기업의 모든 도구에 안전하게 접근할 수 있습니다.

### 사용자 역할

| 역할 | 설명 | 주요 기능 |
|------|------|----------|
| **ADMIN** | 시스템 관리자 | 도구/데이터소스/서버 관리, 승인/거부, 권한 부여, 모든 기능 접근 |
| **USER** | 일반 사용자 | 허용된 도구 조회 및 실행, API 키 생성, 플레이그라운드 사용 |

### 접속 방법

1. 웹 브라우저에서 UMG 주소(예: `https://umg.company.com`)에 접속합니다.
2. 관리자가 발급한 이메일과 비밀번호로 로그인합니다.

## 2. 로그인 및 인증

### 웹 UI 로그인

1. 로그인 페이지에서 **이메일**과 **비밀번호**를 입력합니다.
2. **로그인** 버튼을 클릭합니다.
3. 로그인 성공 시 대시보드로 이동합니다.

> JWT 토큰은 자동으로 관리됩니다. 세션이 만료되면 다시 로그인해야 합니다.

### API 키 인증 (에이전트용)

AI 에이전트는 API 키를 사용하여 MCP 엔드포인트에 인증합니다.

```
X-API-Key: umg_key_a1b2c3d4e5f6...
```

API 키 생성 방법은 [8. API 키 관리](#8-api-키-관리)를 참조하세요.

## 3. 대시보드

로그인 후 첫 화면인 대시보드에서는 시스템 전체 현황을 한눈에 확인할 수 있습니다.

**표시 정보:**

- **등록된 도구 수**: 전체, 타입별(N8N, CUBE_JS, AWS_REMOTE) 분류
- **활성 API 키 수**: 현재 유효한 API 키 개수
- **최근 24시간 요청 수**: 도구 실행 요청 통계
- **오류율**: 최근 도구 실행 실패 비율
- **최근 활동**: 시스템 전체의 최근 도구 실행 이력

## 4. 도구 관리

### 도구 목록 조회

왼쪽 사이드바에서 **도구** 메뉴를 클릭합니다.

**필터링 옵션:**
- **상태**: 전체 / 승인됨(APPROVED) / 대기중(PENDING) / 거부됨(REJECTED)
- **타입**: 전체 / N8N / CUBE_JS / AWS_REMOTE
- **검색**: 도구 이름 또는 설명으로 검색

### 도구 생성 (관리자)

1. 도구 목록 페이지에서 **도구 등록** 버튼을 클릭합니다.
2. 필수 정보를 입력합니다:

| 필드 | 설명 | 예시 |
|------|------|------|
| 이름 | 도구의 고유 식별자 | `production-metrics` |
| 설명 | 도구의 기능 설명 | `MES 시스템에서 생산 라인 지표를 조회합니다` |
| 타입 | 도구 백엔드 타입 | N8N / CUBE_JS / AWS_REMOTE |
| 연결 설정 | 백엔드 연결 정보 (JSON) | URL, 토큰 등 |
| 인증 방식 | 백엔드 인증 방법 | NONE / BEARER / API_KEY |
| 입력 스키마 | 도구 파라미터 정의 (JSON Schema) | `{"type":"object","properties":{...}}` |
| 응답 매핑 규칙 | 응답 데이터 필터링 규칙 | `$.data.metrics[*]` |
| 공개 여부 | 모든 사용자 접근 허용 | 예 / 아니오 |
| 멱등성 여부 | 동일 입력 시 안전한 반복 실행 | 예 / 아니오 |

3. **저장** 버튼을 클릭합니다.

> 새로 생성된 도구는 **PENDING** 상태입니다. 관리자가 승인해야 사용할 수 있습니다.

### 도구 승인/거부 (관리자)

1. 도구 상세 페이지에서 연결 설정과 입력 스키마를 검토합니다.
2. **승인** 또는 **거부** 버튼을 클릭합니다.
3. 승인된 도구만 AI 에이전트가 `tools/list`로 조회하고 `tools/call`로 실행할 수 있습니다.

## 5. Cube.js 데이터소스 관리

> 관리자 전용 기능입니다.

Cube.js 시맨틱 레이어가 연결할 데이터베이스를 등록하고 관리합니다. 왼쪽 사이드바에서 **Cube.js** 메뉴를 클릭합니다.

### 데이터소스 탭

#### 데이터소스 등록

1. **데이터소스** 탭에서 **새 데이터소스** 버튼을 클릭합니다.
2. 연결 정보를 입력합니다:

| 필드 | 설명 | 필수 |
|------|------|------|
| 이름 | 데이터소스 고유 이름 | 예 |
| 설명 | 데이터소스 용도 설명 | 아니오 |
| DB 타입 | 데이터베이스 종류 | 예 |
| 호스트 | DB 서버 주소 | 예 |
| 포트 | DB 서버 포트 | 예 |
| 데이터베이스 | DB 이름 | 예 |
| 사용자명 | DB 접속 계정 | 예 |
| 비밀번호 | DB 접속 비밀번호 | 예 |

**지원되는 DB 타입:**

| DB 타입 | 기본 포트 | 설명 |
|---------|----------|------|
| PostgreSQL | 5432 | 범용 관계형 데이터베이스 |
| MySQL | 3306 | 범용 관계형 데이터베이스 |
| BigQuery | 443 | Google Cloud 데이터 웨어하우스 |
| Redshift | 5439 | AWS 데이터 웨어하우스 |
| Snowflake | 443 | 클라우드 데이터 플랫폼 |
| ClickHouse | 8123 | 고성능 분석 데이터베이스 |

3. **저장** 버튼을 클릭합니다.

> 비밀번호를 포함한 연결 정보는 AES-256으로 암호화되어 저장됩니다.

#### 연결 테스트

1. 데이터소스 목록에서 테스트할 데이터소스의 **테스트** 버튼을 클릭합니다.
2. 시스템이 실제로 DB에 연결을 시도합니다.
3. 결과:
   - **성공**: 연결 응답 시간과 DB 버전이 표시됩니다.
   - **실패**: 오류 메시지가 표시되며 데이터소스 상태가 ERROR로 변경됩니다.

#### 테이블 인트로스펙션

1. 데이터소스 목록에서 **테이블 조회** 버튼을 클릭합니다.
2. DB에 연결하여 사용 가능한 테이블과 컬럼 정보를 조회합니다.
3. 이 정보를 참고하여 Cube.js 스키마를 작성할 수 있습니다.

#### 데이터소스 수정/삭제

- **수정**: 데이터소스 목록에서 해당 행의 **수정** 버튼을 클릭합니다.
- **삭제**: **삭제** 버튼을 클릭합니다. 연결된 스키마가 있으면 함께 삭제됩니다.

## 6. Cube.js 스키마 관리

> 관리자 전용 기능입니다.

Cube.js 시맨틱 레이어의 모델(큐브, 지표, 차원)을 정의하고 관리합니다.

### 스키마 탭

#### 스키마 생성

1. **스키마** 탭에서 **새 스키마** 버튼을 클릭합니다.
2. 정보를 입력합니다:

| 필드 | 설명 | 필수 |
|------|------|------|
| 데이터소스 | 연결할 데이터소스 선택 | 예 |
| 이름 | 스키마 고유 이름 | 예 |
| 설명 | 스키마 용도 설명 | 아니오 |
| 스키마 정의 | JSON 형식의 큐브 모델 정의 | 예 |

**스키마 정의 형식 (JSON):**

```json
{
  "cube": "Orders",
  "measures": {
    "count": {
      "type": "count"
    },
    "totalAmount": {
      "sql": "amount",
      "type": "sum"
    }
  },
  "dimensions": {
    "status": {
      "sql": "status",
      "type": "string"
    },
    "createdAt": {
      "sql": "created_at",
      "type": "time"
    }
  }
}
```

**필수 필드:**
- `cube`: 큐브 이름 (필수)
- `measures` 또는 `dimensions`: 최소 하나 이상 정의 필요

3. **저장** 버튼을 클릭합니다.

> 새 스키마는 **DRAFT** 상태로 생성됩니다.

#### 스키마 상태 관리

| 상태 | 설명 | 전환 액션 |
|------|------|----------|
| **DRAFT** | 초안 상태. AI 에이전트에 노출되지 않음 | **활성화** 클릭으로 ACTIVE 전환 |
| **ACTIVE** | 활성 상태. 메타 API로 조회 가능 | **아카이브** 클릭으로 ARCHIVED 전환 |
| **ARCHIVED** | 보관 상태. 메타 API에서 제외됨 | **활성화** 클릭으로 다시 ACTIVE 전환 가능 |

#### 유효성 검증

1. 스키마 목록에서 **검증** 버튼을 클릭합니다.
2. 시스템이 스키마 정의를 분석합니다:
   - JSON 구문이 유효한지 확인
   - `cube` 필드가 존재하는지 확인
   - `measures` 또는 `dimensions` 중 하나 이상 존재하는지 확인
3. 검증 결과가 표시됩니다.

> 스키마를 활성화하려면 유효성 검증을 반드시 통과해야 합니다.

#### 버전 관리

- 스키마 정의(schemaDefinition)를 수정하면 **버전이 자동으로 증가**합니다.
- 이름이나 설명만 변경하면 버전이 유지됩니다.
- 현재 버전 번호는 스키마 목록에서 확인할 수 있습니다.

### 메타 탐색기 탭

활성화된 모든 스키마의 메타 정보를 한눈에 확인할 수 있습니다.

**표시 정보:**
- 큐브 이름
- 스키마 이름
- 정의된 지표(Measures) 목록
- 정의된 차원(Dimensions) 목록

이 정보는 AI 에이전트가 Cube.js 도구를 호출할 때 사용할 수 있는 지표와 차원을 확인하는 데 유용합니다.

## 7. AWS MCP 서버 관리

> 관리자 전용 기능입니다.

AWS에 호스팅된 원격 MCP 서버(Redshift, S3 Tables, CloudWatch, Cost Analysis 등)를 등록하고 관리합니다. 왼쪽 사이드바에서 **AWS MCP** 메뉴를 클릭합니다.

### 서버 탭

#### 서버 등록

1. **서버** 탭에서 **새 서버** 버튼을 클릭합니다.
2. 서버 정보를 입력합니다:

| 필드 | 설명 | 필수 |
|------|------|------|
| 이름 | 서버 고유 이름 | 예 |
| 설명 | 서버 용도 설명 | 아니오 |
| 엔드포인트 URL | MCP 서버의 HTTPS URL | 예 |
| 리전 | AWS 리전 코드 | 예 |
| 서비스 | AWS 서비스명 (기본: execute-api) | 아니오 |
| 인증 방식 | IAM_KEY 또는 IAM_ROLE | 예 |

**인증 방식별 추가 필드:**

| 인증 방식 | 추가 필드 |
|-----------|----------|
| **IAM_KEY** | Access Key ID, Secret Access Key |
| **IAM_ROLE** | Role ARN (+ 선택적으로 Access Key ID, Secret Access Key) |

3. **저장** 버튼을 클릭합니다.

> AWS 자격 증명은 AES-256으로 암호화되어 저장됩니다.

#### 연결 테스트

1. 서버 목록에서 **테스트** 버튼을 클릭합니다.
2. 시스템이 AWS SigV4 서명을 사용하여 `initialize` MCP RPC를 전송합니다.
3. 결과:
   - **성공**: MCP 프로토콜 버전과 서버 이름이 표시됩니다.
   - **실패**: 오류 메시지가 표시되며 서버 상태가 ERROR로 변경됩니다.

#### 도구 동기화

AWS MCP 서버에서 사용 가능한 도구를 자동으로 발견하고 UMG에 등록합니다.

1. 서버 목록에서 **동기화** 버튼을 클릭합니다.
2. 시스템이 `tools/list` MCP RPC를 전송하여 도구 목록을 조회합니다.
3. 결과:
   - **새 도구**: `PENDING` 상태로 자동 등록됩니다. 관리자가 승인해야 사용 가능합니다.
   - **기존 도구**: 설명과 입력 스키마가 최신 정보로 업데이트됩니다.
4. 동기화 결과 요약이 표시됩니다:
   - 발견된 도구 수
   - 새로 생성된 도구 수
   - 업데이트된 도구 수

> 동기화로 생성된 도구는 `awsServerId`를 통해 원본 서버와 연결됩니다. 도구 실행 시 서버 레지스트리에서 자격 증명을 자동으로 가져옵니다.

### 동기화 이력 탭

서버별 도구 동기화 실행 이력을 확인할 수 있습니다.

**표시 정보:**

| 필드 | 설명 |
|------|------|
| 상태 | SUCCESS / FAIL |
| 발견된 도구 | 서버에서 발견된 도구 수 |
| 생성된 도구 | 새로 등록된 도구 수 |
| 업데이트된 도구 | 기존 도구가 업데이트된 수 |
| 오류 메시지 | 실패 시 상세 오류 내용 |
| 실행 일시 | 동기화가 수행된 시간 |

## 8. API 키 관리

API 키는 AI 에이전트(Cursor, Claude Desktop 등)가 MCP 엔드포인트에 인증할 때 사용합니다.

### API 키 생성

1. 왼쪽 사이드바에서 **API 키** 메뉴를 클릭합니다.
2. **API 키 생성** 버튼을 클릭합니다.
3. 정보를 입력합니다:

| 필드 | 설명 |
|------|------|
| 이름 | 키의 용도를 식별하는 이름 (예: "Cursor IDE - 개발 노트북") |
| 분당 요청 제한 | 이 키로 분당 실행할 수 있는 최대 요청 수 |
| 만료일 | 키의 만료 일시 (선택) |

4. **생성** 버튼을 클릭합니다.
5. **원본 키가 한 번만 표시됩니다.** 반드시 안전한 곳에 복사하여 저장하세요.

> API 키는 SHA-256 해시로 저장되므로, 원본 키를 분실하면 복구할 수 없습니다. 새 키를 생성해야 합니다.

### API 키 관리

- **비활성화**: 키를 일시적으로 사용 중지합니다.
- **삭제**: 키를 영구 삭제합니다.
- **교체**: 기존 키를 삭제하고 새 키를 생성합니다.

## 9. 권한 관리

> 관리자 전용 기능입니다.

도구별로 사용자의 접근 권한을 관리합니다.

### 권한 부여

1. 왼쪽 사이드바에서 **권한** 메뉴를 클릭합니다.
2. **권한 부여** 버튼을 클릭합니다.
3. 대상 도구와 사용자를 선택합니다.
4. 접근 수준을 선택합니다: **EXECUTE** (실행 권한)
5. **저장** 버튼을 클릭합니다.

### 권한 확인

- 도구별 또는 사용자별로 권한 목록을 필터링할 수 있습니다.
- 공개 도구(isPublic=true)는 별도의 권한 없이 모든 인증된 사용자가 접근할 수 있습니다.

## 10. 감사 로그

모든 도구 실행 이력이 자동으로 기록됩니다.

### 감사 로그 조회

1. 왼쪽 사이드바에서 **감사 로그** 메뉴를 클릭합니다.

**필터링 옵션:**

| 필터 | 설명 |
|------|------|
| 사용자 | 특정 사용자의 실행 이력만 조회 |
| 도구 | 특정 도구의 실행 이력만 조회 |
| 상태 | SUCCESS / FAILURE / ERROR |
| 기간 | 시작일 ~ 종료일 |

**표시 정보:**
- 실행 일시
- 사용자 이름
- 도구 이름
- 실행 상태
- 실행 소요 시간
- 입력 파라미터 (JSON)
- 에러 메시지 (실패 시)

## 11. AI 에이전트 연동

### Cursor IDE 연동

1. API 키를 생성합니다 ([8. API 키 관리](#8-api-키-관리) 참조).
2. 프로젝트 루트에 `.cursor/mcp.json` 파일을 생성합니다:

```json
{
  "mcpServers": {
    "umg": {
      "command": "npx",
      "args": [
        "-y", "umg-bridge",
        "--server-url", "https://umg.company.com",
        "--api-key", "umg_key_xxxxx"
      ]
    }
  }
}
```

3. Cursor IDE를 재시작합니다.
4. 채팅에서 UMG에 등록된 도구를 호출할 수 있습니다.

### Claude Desktop 연동

1. API 키를 생성합니다.
2. `claude_desktop_config.json` 파일에 추가합니다:

```json
{
  "mcpServers": {
    "umg": {
      "command": "npx",
      "args": ["-y", "umg-bridge"],
      "env": {
        "UMG_SERVER_URL": "https://umg.company.com",
        "UMG_API_KEY": "umg_key_xxxxx"
      }
    }
  }
}
```

3. Claude Desktop을 재시작합니다.

### 직접 API 호출

API 키를 사용하여 MCP 프로토콜로 직접 통신할 수 있습니다.

**1단계: 초기화**

```bash
curl -X POST https://umg.company.com/api/mcp \
  -H "X-API-Key: umg_key_xxxxx" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "id": 1,
    "method": "initialize",
    "params": {
      "protocolVersion": "2024-11-05",
      "capabilities": {},
      "clientInfo": {"name": "custom-agent", "version": "1.0"}
    }
  }'
```

**2단계: 도구 목록 조회**

```bash
curl -X POST https://umg.company.com/api/mcp \
  -H "X-API-Key: umg_key_xxxxx" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "id": 2,
    "method": "tools/list",
    "params": {}
  }'
```

**3단계: 도구 실행**

```bash
curl -X POST https://umg.company.com/api/mcp \
  -H "X-API-Key: umg_key_xxxxx" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "id": 3,
    "method": "tools/call",
    "params": {
      "name": "production-metrics",
      "arguments": {
        "lineId": "LINE-003",
        "metric": "temperature"
      }
    }
  }'
```

## 12. 문제 해결

### 로그인 실패

| 증상 | 원인 | 해결 |
|------|------|------|
| "Invalid credentials" | 이메일 또는 비밀번호 오류 | 올바른 자격 증명으로 재시도 |
| "Account disabled" | 계정 비활성화 | 관리자에게 계정 활성화 요청 |

### 도구 실행 실패

| 증상 | 원인 | 해결 |
|------|------|------|
| "Tool not found" | 도구가 존재하지 않거나 미승인 | 관리자에게 도구 승인 요청 |
| "Access denied" | 실행 권한 없음 | 관리자에게 EXECUTE 권한 요청 |
| "Rate limit exceeded" | 분당 요청 한도 초과 | 잠시 대기 후 재시도 |
| "Tool execution failed" | 백엔드 시스템 오류 | 감사 로그에서 상세 오류 확인 |

### 데이터소스 연결 실패

| 증상 | 원인 | 해결 |
|------|------|------|
| "Connection refused" | DB 서버 접근 불가 | 호스트/포트 확인, 방화벽 규칙 확인 |
| "Authentication failed" | DB 인증 실패 | 사용자명/비밀번호 확인 |
| "Unknown database" | DB 이름 오류 | 데이터베이스 이름 확인 |
| "Connection timeout" | 네트워크 지연 | 네트워크 연결 확인 |

### AWS MCP 서버 연결 실패

| 증상 | 원인 | 해결 |
|------|------|------|
| "403 Forbidden" | AWS 자격 증명 오류 | Access Key / Secret Key 확인 |
| "SignatureDoesNotMatch" | SigV4 서명 오류 | 리전과 서비스 설정 확인 |
| "Connection timeout" | 엔드포인트 접근 불가 | URL 확인, VPC/보안 그룹 확인 |
| "Invalid JSON-RPC" | MCP 프로토콜 불일치 | 서버가 MCP 프로토콜을 지원하는지 확인 |

### 도구 동기화 실패

| 증상 | 원인 | 해결 |
|------|------|------|
| "initialize failed" | 서버 초기화 실패 | 연결 테스트 먼저 수행 |
| "tools/list failed" | 도구 목록 조회 실패 | 서버 상태 확인 |
| "Sync history: FAIL" | 동기화 중 오류 발생 | 동기화 이력에서 오류 메시지 확인 |
